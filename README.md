# Colab Training (Token Flow)

This folder contains a Google Colab notebook template for VisoLabel cloud handoff.

## Expected flow

1. In VisoLabel, click `Upload Dataset to Colab`.
2. The app packages dataset + settings into a zip bundle.
3. The desktop app generates a random password, encrypts the zip with AES-256-GCM, and uploads only the encrypted file to object storage.
4. The app shows a compact encrypted Colab token containing the password, server token, and compact non-secret encryption metadata. New encrypted tokens use `VL1.<base64url-json-array>` with `[server_token, password, metadata]` or `[server_token, password, api_url, metadata]`. The temporary download URL is resolved by the notebook through the API.
5. In Colab, run `VisoLabel_Train_From_Token.ipynb` and paste the token.
6. The notebook downloads the encrypted bundle, decrypts it locally in Colab, starts training, and uploads checkpoints back to S3 (through API-issued presigned URLs).

Treat the token as a secret: anyone with it can download and decrypt the bundle until the presigned URL expires.

## Notebook GUI

The notebook is designed to show as little code as possible. It has a short
usage section and one Colab form cell named `Launch VisoLabel Training GUI`.
Running that cell opens an ipywidgets interface with:

- Token input and optional API URL override.
- Bundle preview for task, model, epochs, batch size, image size, class count, and image count.
- `Run full pipeline` for normal use.
- Separate resolve, train, and upload buttons for troubleshooting.
- Progress, status cards, and live logs for download, decryption, extraction, training, and upload.

## Required API endpoints

The notebook expects these endpoints for checkpoint upload callbacks and legacy unencrypted tokens:

- `POST /api/v1/colab/resolve-token/`
- `POST /api/v1/colab/request-checkpoint-upload/`
- `POST /api/v1/colab/finalize-run/` (optional but recommended)

The desktop app uses:

- `POST /api/v1/colab/request-upload/`
- `POST /api/v1/colab/finalize-upload/`

For large bundles, the desktop app requests multipart upload mode. If
`request-upload` returns `upload_method: "MULTIPART"` or `S3_MULTIPART` with
presigned part URLs, the app uploads parts in parallel and sends the returned
ETags as `parts` to `finalize-upload`.

For encrypted tokens, `request-upload` stores the non-secret `encryption`
metadata. The notebook can use the compact metadata embedded in new tokens, but
`resolve-token` should still return a temporary `bundle_url`/`download_url` plus
that `encryption` metadata when called with the server token from the Colab
token. The password is generated by the desktop app and is never sent to the API.

## Bundle contract

After decryption, each bundle includes:

- `training_bundle.json`
- `dataset/` (task-specific layout)

`training_bundle.json` stores immutable training settings selected in the app.
